# VIN GO — Дорожная карта

Статус продукта подбора автозапчастей по VIN/госномеру. Обновляется по мере работы.

## Архитектура
- Монорепо на Bun: `backend` (Hono + Prisma + PostgreSQL), `webapp` (Vite + React + TanStack + shadcn), `mobile` (Expo, standalone), `packages/contracts` (Zod), `website` (Astro).
- Единое backend-ядро для веба, Telegram-бота и мобильного.
- Слои провайдеров (мок сейчас → реальные API позже без переписывания): `CatalogProvider` (VIN→OEM), `SupplierProvider` (номер→предложения), `PlateProvider` (госномер→VIN), `PaymentProvider` (мок + ЮKassa).
- Деньги в копейках (int) + валюта. Локальная БД — нативный PostgreSQL 18 (:5432).

## ✅ Реализовано

### Этап 1 — Поиск
- VIN → авто, госномер → VIN (с нормализацией латиницы→кириллицы), поиск запчастей, предложения.
- Раскладка по 3 тирам «Эконом / Оптимальный / Оригинал» (`tiering.ts`).

### Этап 2 — Кабинет (БД + авторизация)
- Авторизация (refresh-токены, `requireAuth`).
- Гараж (сохранённые авто), корзина (цена авторитетна с сервера), заказы.
- Жизненный цикл заказа: `DRAFT → PLACED → PAID → PROCESSING → READY → COMPLETED` (+ `CANCELLED`), карта переходов в контрактах.

### Этап 3 — Оплата
- Мок работает end-to-end (`/pay`), `YooKassaPaymentProvider` готов (активируется `YOOKASSA_*`).
- Идемпотентность, авто-переход заказа в PAID, webhook.

### Этап 4 — Telegram-бот
- Long-polling, без новых зависимостей, переиспользует `CatalogService`. Поиск по VIN/госномеру → тиры.

### Этап 5 — Мобильное (Expo)
- MVP: вход + поиск (VIN/госномер) + тиры. Запуск на телефоне через Expo Go.

### Тесты
- Контракты, backend unit, интеграция на живой БД; typecheck всех пакетов и сборка веба — зелёные.

- [x] Привязка корзины к авто из гаража (`POST /api/cart/vehicle`, селектор в корзине).

- [ ] **Возвраты/частичная отмона** — отложено: требует реального API возвратов ЮKassa
      (мок-возврат вводил бы в заблуждение). Делать вместе с боевой оплатой (раздел 3).

### 3. Оплата (до боевого режима)
- [x] **Возвраты** (`POST /api/payments/refund`, модель `Refund`, статус заказа `REFUNDED`),
      мок работает end-to-end, в ЮKassa — `POST /v3/refunds`. Закрывает и пункт 2c.
- [x] **Защита webhook**: телу не доверяем — статус перепроверяется через API (`getStatus`/
      `getRefundStatus`); при неудаче верификации бросаем ошибку (5xx → ретрай ЮKassa), а не
      проводим платёж. Обрабатывается `refund.succeeded`. Опц. IP-allowlist (`YOOKASSA_WEBHOOK_IP_ALLOWLIST`).
- [x] **СБП**: выбор способа оплаты (Карта/СБП), проброс в ЮKassa (`payment_method_data`).
- [x] **Reconcile** зависших платежей/возвратов: cron `payments:reconcile` (страховка от
      потерянного webhook — платёж не зависнет в PENDING при списанных деньгах).
- [x] **Уведомление при оплате**: `PLACED→PAID` и `REFUNDED` шлют push/Telegram владельцу.
- [x] **Юнит-тесты `YooKassaPaymentProvider`** (мокнутый fetch: суммы, СБП, статусы, чек, ошибки).
- [x] **Фискализация (54-ФЗ)**: чек из позиций заказа + email клиента в платёж и возврат,
      включается `YOOKASSA_VAT_CODE`. Требует подключённой кассы/ОФД в ЛК ЮKassa.
- [ ] **Боевая ЮKassa**: реальный аккаунт, ключи `YOOKASSA_*`, регистрация webhook, sandbox-прогон.
      Чеклист включения — `docs/DEPLOYMENT.md` (раздел «ЮKassa Payments Go-Live»).
